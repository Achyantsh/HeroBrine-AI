# HeroBrine AI Database Design

## Overview

HeroBrine AI revolves around a single core entity called **Commitment**.

A Commitment represents anything the user has promised, planned, or is responsible for completing.

Examples include:

- Assignment
- Interview
- Exam
- Meeting
- Flight
- Bill Payment
- Health Routine
- Project Milestone
- Personal Goal

Everything in HeroBrine is built around this entity.

---

# Core Entity

## Commitment

Represents a single actionable obligation.

### Fields

| Field | Description |
|--------|-------------|
| id | Unique identifier |
| title | Short title of the commitment |
| description | Detailed description |
| type | Assignment, Interview, Meeting, Exam, Bill, Goal, etc. |
| source | Manual, Email, WhatsApp, PDF, Calendar, Voice |
| priority | Low, Medium, High, Critical |
| status | Pending, In Progress, Completed, Missed, Cancelled |
| deadline | Due date and time |
| estimated_duration | Estimated effort required |
| created_at | Creation timestamp |
| updated_at | Last modified timestamp |

---

# Why only one entity?

Instead of building separate tables for:

- Assignments
- Meetings
- Interviews
- Bills

HeroBrine treats them all as Commitments.

This keeps the system flexible and makes AI reasoning much simpler.

---

# Future Entities

These are intentionally **not implemented in Version 1**.

## ExecutionPlan

Stores AI-generated execution strategies.

Example:

- Break task into steps
- Suggested working sessions
- Estimated completion timeline

---

## Attachment

Stores uploaded files.

Examples:

- PDFs
- Images
- Screenshots
- Documents

---

## Notification

Stores reminders and alerts generated by HeroBrine.

---

## Dependency

Represents relationships between commitments.

Example:

Submit PPT
↓

Project Demo

The Project Demo commitment depends on the PPT Submission commitment.

---

## RiskAnalysis

Stores AI-generated risk assessments.

Examples:

- High risk of missing deadline
- Medium workload
- Schedule conflict

Risk is generated by AI and is **not** part of the Commitment entity.

---

# Database Design Principles

1. One universal Commitment model.
2. AI-generated information lives outside the Commitment table.
3. Computed values (time left, risk score, completion percentage) are not stored.
4. Future features should extend the database instead of modifying the Commitment entity.
